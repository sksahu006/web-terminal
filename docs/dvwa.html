How DVWA fits into the Virtual Workspace Platform

Inside this major project workspace, DVWA is used as a challenge room environment. When a learner spawns the web-basics room:

Dual-Container Topology

The platform launches an attacker container (carrying tools like curl, netcat, etc.) and a private DVWA target container connected over the same Docker bridge network.

Private Target Protection

The DVWA target container does not expose any host ports. This prevents lateral attacks and limits exploitation access exclusively to the student's assigned attacker container.

Target Injection

The attacker container is provisioned with a $TARGET_URL environment variable, enabling learners to immediately query and attack the DVWA instance directly from their terminal session.

Vulnerability Category

Command Injection

Subdirectory: DVWA/vulnerabilities/exec/

HIGH SEVERITY

Command injection occurs when an application passes unsafe user-supplied data to a system shell. In this scenario, the operating system commands are executed with the privileges of the vulnerable application.

PHP Source Code comparison

<?php
// Low level source code snippet
if( isset( $_POST[ 'Submit' ]  ) ) {
    $target = $_REQUEST[ 'ip' ];
    if( stristr( php_uname( 's' ), 'Windows NT' ) ) {
        $cmd = shell_exec( 'ping  ' . $target );
    } else {
        $cmd = shell_exec( 'ping  -c 4 ' . $target );
    }
    $html .= "<pre>{$cmd}</pre>";
}
?>

Vulnerability Flaw

No sanitation filters are applied on the target input. The variable is concatenated directly into the ping command and passed to shell_exec. Attackers append logical operators (like ;, &, or ||) followed by arbitrary shell commands to run payloads.

Prevention & Remediation

Instead of shell command concatenation, validate that inputs represent an IP address format strictly (whitelisting numeric characters and periods). Alternatively, escape parameters using escapeshellarg() or avoid shell functions entirely in favor of native programming language APIs.

DVWA Security Levels Mechanics

Understanding security modes toggled via user session cookies

LEVEL: LOW

Totally Vulnerable

No security measures are implemented. Parameter inputs are passed straight into SQL queries or shell executors. Suitable for confirming baseline exploitation and executing standard proof-of-concept payloads.

LEVEL: MEDIUM

Naive Filters (Blacklists)

Implements minor sanitization steps, such as regex filters replacing common character patterns (e.g. replacing <script> with empty spaces). Easily bypassed using capitalization tricks or nested injection schemes.

LEVEL: HIGH

Strict Filters / Blacklists

Applies comprehensive blacklist logic. Filters out multiple command concatenations or restricts characters stringently. Still bypassable using obscure protocols, alternative syntax formats, or out-of-band methods.

LEVEL: SECURE

Secure Implementations

Utilizes robust programming standards: SQL prepared statements (eliminating code injection vectors), strict input whitelisting, output context-aware encoding, and mandatory anti-CSRF token verification checks.
