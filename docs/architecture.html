Module Structure & Architecture Breakdown

FE React Frontend

Single-Page React App built with Vite and Tailwind v4. Features standard cookie/JWT auth storage, dynamic loading indicators for container startup, dashboard for launching labs, and embedded iframe wrappers targeting the dynamically exposed container port.

BE FastAPI Backend Server

Asynchronous Python backend running on Uvicorn. Decoupled into routers, models, schemas, and services. Uses async database sessions and validates permissions prior to issuing commands to Docker.

DO Docker Orchestrator

A dedicated service wrapper (`ContainerManager`) querying the local Docker socket. Enforces strict limits: custom CPU weights, memory ceilings, PID execution counts (256), and cleans up containers instantly.

LA Challenge Labs

Provides disposable cybersecurity/coding environments. Templates describe configurations (terminal tools, isolated per-session target networks, minimal CPU/memory footprints) with flag challenges checked in SQLite database on submission. Current DVWA-style catalog: SQL Injection, Command Injection, Reflected XSS, Stored XSS, and Unrestricted File Upload, plus the original Web Basics room.

Workspace Container Lifecycle

Lifecycle flow coordinates user requests with Docker SDK client inside container_manager.py

STEP 01

Auth & Authorization

Client requests `/workspace/start` providing a JWT. Backend validates JWT, verifies user status, and checks if an active container already exists for this user.

STEP 02

Limit Retrieval

DB queries user limits: CPU shares, Memory MB limits, Disk limits, and maximum lease run duration. Standard defaults are loaded if no custom limits exist.

STEP 03

Container Creation

Python Docker SDK issues a run command on `/var/run/docker.sock` applying `mem_limit`, `nano_cpus`, and `pids_limit=128`. Standalone terminal labs join the shared `workspace-network`; web-target labs instead get a freshly created, per-session bridge network (`lab-{user}-{slug}-net`) so no other session's target container is reachable.

STEP 04

Port Resolution

Host port is assigned dynamically by Docker daemon. Backend polls container state, extracts the host port, updates the `workspaces` table with `status="running"`.

STEP 05

Auto-Kill & Expiry

Client requests status periodically. If `expires_at` is exceeded, or user triggers stop, backend invokes container stop and force-removes the Docker resources.

Security Isolation Constraints (Built-in)

To protect the host system, spawned containers are forced with privileged=False. No access to the Docker socket is mounted inside user workspace containers. Limits are hard-enforced at the container level by the Linux kernel CGroups via Docker engine, preventing denial of service attacks by resource exhaustion.
